11 - System Initialization

When powering on a system, a bootloader will decompress the Linux kernel image and load it into memory. However, the kernel itself is not a fully functioning system. The initialization process refers to starting various services and applications after boot to bring a system into its desired functioning state.

A traditional Unix system used the init process, which started running as the very first process with the PID=1. The binary executable for the init is located at /sbin/init. The init proceeds to execute any other programs and services that were configured to load. The init process is the "parent" of all processes in your system. If a process is "orphaned" (parent process is "terminated", crashed, or "killed"), init will automatically adopt the process.

Linux adopted the System V init, which works simply the same way it does in commercial versions of Unix and BSD. It remained this way for many years; however, as of the 2010s, most Linux distributions have switched away from init. The sysvinit style scripts still exist on some systems as a way of a backward compatibility layer. System runlevels discussed here are generally considered "legacy" and obsolete. You may never have to use runlevels and init; however, you may run across documentation and information that may refer to these or use the language due to historical reasons. There are edge cases of Linux distros that advertise themselves as "systemd-free" and may still be using sysvinit or equivalent.

Examples of other system initialization and service management methods include systemd, launchd, and OpenRC.

System Runlevel (Obsolete in most Linux Distros)

The concept of runlevels has been in use in System V-style initialization since the 1980s. It is a method of defining groups of software and services to execute upon system startup. For example, a runlevel can be defined as "Single User Mode Without Networking", and another as "Multi-user Mode With Networking". Windows has a similar concept; if you are familiar with the Windows "Safe Mode", "Safe Mode with Networking", and "Normal Mode", etc.

Linux adopted this method of initialization, and there are baseline specifications for this. However, Linux distros pick and choose how to define and implement the runlevels. For more information, refer to Wikipedia - Runlevel.

The following is the Linux standard baseline specifications from the Wikipedia - Runlevel article:

ID Name Description
0 Off Turns off the device.
1 Single-user mode Mode for administrative tasks.
2 Multi-user mode Does not configure network interfaces and does not export network services.
3 Multi-user mode with networking Starts the system normally.
4 Not used/user-definable For special purposes.
5 Full mode Same as runlevel 3 + display manager.
6 Reboot Reboots the device.

In order to start a system in a specific pre-defined runlevel, use the init command. Here are some examples using the table above:

# will drop the system into "Single-user mode" with only the root user and minimal services (equivalent to "safe mode" or "recovery mode")
sudo init 1  

# This is the "normal" operations mode for your Ubuntu server
sudo init 3

# This will reboot the system
sudo init 6

# Theoretically, this should tell you what system runlevel you are running on. Give it a try to see if this works on your system!
runlevel

Please note that Linux distros may define their own runlevel IDs and may not follow the above specification. You may also customize these modes or add other modes yourself. The sysvinit has the runlevels defined in the /etc/inittab file. The actual executable is located at /sbin/init. For more information, refer to man 8 runlevel.

The systemd System and Service Manager

The most common modern initialization system is currently systemd. Ubuntu and Debian have been using systemd since 2014. Currently, systemd has completely replaced sysvinit, and the /etc/inittab file may not exist by default.

One of the biggest advantages of systemd over the older sysvinit is parallelization. The systemd can take advantage of modern multi-threaded and multi-core systems to improve performance and reduce boot times. Additionally, you can define and manage service dependencies. For example, a network-based service may need to wait for a particular network interface to become available before starting its process. In traditional init, the system will be stuck on boot waiting for the network interface to initialize. In case of a failure, the system potentially waits a long time for a timeout to occur before the system can proceed with loading the rest of the services. The systemd initialization can continue to load other services that do not have that dependency and bring the system to a usable state much faster.

The systemd is not just an initialization utility, but it has a wide range of utilities and tools aiming to unify system and service management. It has provided a replacement for many older and legacy daemons and services in Linux. Of course, this may break application compatibility with Unix systems; however, having a unified and centralized system for managing all system services and processes is an advantage for users and developers. Some examples of the core and ancillary systemd components include:

  • systemd, a system and service manager for Linux
  • systemctl (see man systemctl) controls the systemd system and service manager
  • systemd-analyze: control the systemd system and service manager, also used to access special functions useful for advanced system manager debugging.
  • journald: facilities for event logging
  • lacaled: for managing keyboard layout and locale formats such as date and time, and language
  • logind: for managing user logins
  • networkd: network renderer for managing network configuration (you have seen this one!)
  • resolved: Local DNS resolver
  • systemd-bosd: relatively new addition, if you missed Microsoft Windows style "Blue Screen of Death"
  • timedated: for managing and controlling local time and date
  • timesyncd: an NTP client
  • udevd: a device management facility for the user space. It handles adding and removing devices (for example, USB devices or other hot-swap hardware)

For more information about systemd development and official documentation, visit systemd.io. For more information about systemd, you can also read the Wikipedia - systemd article.

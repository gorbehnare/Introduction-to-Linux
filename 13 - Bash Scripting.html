13 - Bash Scripting

Bash scripts are plain text files that contain a list of bash commands. They are widely used and extremely useful. They can be used for a wide range of purposes, from simplifying complex tasks to automating repetitive tasks.

Executing Files

A file is made executable by chmod +x for the appropriate user or group. Running an executable file is as simple as calling the name of the file in the shell using an absolute or relative path (for example: ./script.sh).

A file can also be executed without having the execute permissions, by passing it to a shell or other program to execute:

  • bash script.sh: a new bash process is started with the current user's permissions and will run commands from inside the file
  • sh script.sh: a new simple shell process is started with the current user's permissions and will run the commands inside the file

In general, Linux does not identify file types by the filename or extension. To specify what program is intended to run the script or executable file in, a "shebang" is used at the very beginning of the file, followed by the path to the program executable that should run the file: #!<path_to_program>. The #! (This really is called a shebang) is often used by systems to figure out which program should run or open a file in place of an extension, when you are executing the file directly, for example, if you double-click on a file in a GUI shell. When you are writing a shell script, you specify the shell program for which you have written the script:

  • #!/bin/bash: This file is intended to be opened in bash.
  • #!/bin/sh: This file is intended to be opened in the system shell.

File Paths

Please be mindful of specifying paths inside your script. Sometimes it is appropriate to use absolute paths, while other times relative paths are the better choice. Make sure that your script is as generic as possible so that it can be executed in any environment. Additionally, you have to take access and permissions into consideration.

  • Absolute path: /absolute/path/to/a/file
  • Relative path: ./relative/path/to/a/file

Input and Output Redirection

In bash, there are two "standard outputs": stdout (standard output), and stderr (standard error). For redirecting output to another file or program:

  • Use the > character for stdout
  • Use 2> for stderr
  • &> will redirect both
    There are a few more characters with special meaning:
  • < for taking input from a file (instead of keyboard).
    • Standard input (stdin) usually comes from the keyboard or terminal.
  • $? stores the exit status of the last command.
    • Exit status 0 usually indicates success.
    • Other status numbers can have different meanings. Refer to the manual pages of each specific utility to find the exit status and meaning.
    • You can output exit status for your own scripts, or output errors to users when something goes wrong
  • The pipe | character is used to redirect the stdout of the first command into the stdin of the next command
    An example of using exit status:
ls /path/to/some/file  # This is just an example to show how '$?' works
if [ $? -eq 0 ] then   # Note: There are better ways of checking if a path exists without using 'ls'!
   echo “it worked”
 else
    echo “it didn’t work!”
fi

Executing Multiple Commands

  • The ; character is used to indicate the end of one command and the beginning of the next.
    • Commands are executed in order without having any dependence on each other
    • Example: cp ./a* ./letters ; cp ./1* ./numbers (runs these commands one after the other, without caring for results)
  • The & character is used to push a task to the background.
    • Example: cp ./large_files /media/backup_disk & will start copying and will send the job to the background, so you can continue using your terminal without having to wait for the copy operation to finish
    • The bg and fg commands are used to list the tasks running in the background and bring them to the foreground when needed.
  • The && performs a logical AND operation. It will only execute the second command if the first command is successful
    • Example: mkdir new_directory && mv file1 new_directory This will not execute the mv command if mkdir fails.
  • The || performs a logical OR operation. It will only execute the second command if the first command fails
    • Example: cd ./local || echo "Couldn't!!! Are we in the correct location?"

Variables

By default, variables are created in the local process memory, and your program or script will not be able to modify the environmental variables. Local variables are inherited by child processes. If you need to add, remove, or change other environmental variables, there are env, export, set, and unset commands that can be used. For more information about working with local, user, and system variables, see this article:

  • https://www.tecmint.com/set-unset-environment-variables-in-linux/

To set a local variable, simply declare the variable name. In bash, the $ is used to access the information stored inside a variable. Here is an example:

user@hostname:~$ Variable_name=value
user@hostname:~$ echo “The variable is set to $Variable_name”
“The variable is set to value”
user@hostname:~$
user@hostname:~$ export testvar="test"
user@hostname:~$ echo $testvar
test
user@hostname:~$ unset testvar
user@hostname:~$ echo $testvar

user@hostname:~$ whatis env

Positional Parameters

Using parameters and positional parameters in the CLI shell is quite common. As an example in cd directory, you are supplying the parameter directory to cd. Parameters can be anything that the executable needs, such as paths, values, flags, and options. It is up to the developer of the program to use these parameters as required and see fit. These arguments are passed to the program, and you can access them inside your script.

Here is an example of how this works.

  • Consider: ./script.sh argument1 argument2 argument3 ... argumentN
  • Inside your script, the following variables are available:
    • $1 is set to argument1, $2 is argument2 ... $N is argumentN (N is a positive integer referring to the Nth argument in the list)
    • $# is the total number of arguments, so in this example, $# = N
    • $* refers to all input arguments
  • These positional parameters also work with the content of the file when using the < redirection.

Numeric and String Comparison and File Conditionals

By default, symbols trigger a string comparison (the system looks at letters, not the values of numbers). Here is a table of comparisons:

Description Numeric Comp. String Comp.
less than -lt <
greater than -gt >
equal -eq =
not equal -ne !=
less or equal -le
greater or equal -ge

The results of the comparisons are not sent to the stdout, and only an exit status is returned for true=0 or false=1. In the terminal, you can get the exit status from the last command using $?.

user@hostname:~$ a=100
user@hostname:~$ b=200
user@hostname:~$ [ $a -eq $b ]
user@hostname:~$ echo $?
1
user@hostname:~$ # NOTE: True = 0, False = 1
user@hostname:~$ [ $a -gt $b ] || echo "$a is NOT greater than $b“
100 is NOT greater than 200
user@hostname:~$ a=100
user@hostname:~$ b=200
user@hostname:~$ [ $a -eq $b ]
user@hostname:~$ echo $?
1
user@hostname:~$ # NOTE: True = 0, False = 1
user@hostname:~$ [ $a -gt $b ] || echo "$a is NOT greater than $b“

In the above examples, we are using the single square bracket format to evaluate conditions. Comparisons and conditionals can be used on their own using [ ] and [[ ]] or with if and case statements. These are discussed later in this document.

You can also perform arithmetic operations in Bash using (( )) format. This is currently out of scope for us, but if you are interested, you can refer to the GNU Bash Manual's sections on Conditional Constructs and Shell Arithmetic.

Conditional Constructs

There are two formats for conditional constructs in Linux and Unix:

  • Single bracket format [ ] (the POSIX standard format)
    • Will work on POSIX-compliant shells, and therefore portable
  • Double bracket format[[ ]] (the extended format)
    • Supports more advanced features and an extension to the standard [ ] format. These only work in bash, zsh, ksh, and other more advanced shells and do not work in sh. Scripts using the [[ ]] format should start with the correct "shebang", such as #!/bin/bash. The single and double bracket formats also extend to case and if statements.

If Statement

Here is the basic template for the if statement syntax:

# Using Single Brackets
if [ conditions ]; then
        do_something
elif [ another_condition ]; then
        do_something_else
else
        do_more_stuff
fi        # The end of the if statement (if spelled backwards)

# Using Double Brackets
if [[ expression ]]; then
        do_something
elif [[ another_expression ]]; then
        do_something_else
else
        do_more_stuff
fi

Case Statement

case  $variable-name  in
   pattern1)       # $variable-name is compared to find a match
       command1    # if matched, then do this
       ...
       commandN
       ;;          # the ";;" means done here, go to "esac" (exit the case)  
   pattern2 | pattern3 | pattern4)
       command1
       ...
       commandN
       ;;            
   patternN)       
       command1
       ...
       commandN
       ;;
    *)             # equivalent to else (e.g. everything else)
       command1
esac               # The end of case statement (case spelled backward)
case  $variable-name  in
   pattern1)       # $variable-name is compared to find a match
       command1    # if matched, then do this
       ...
       commandN
       ;;          # the ";;" means done here, go to "esac" (exit the case)  
   pattern2 | pattern3 | pattern4)
       command1
       ...
       commandN
       ;;            
   patternN)       
       command1
       ...
       commandN
       ;;
    *)             # equivalent to else (e.g. everything else)
       command1
esac               # The end of case statement (case spelled backward)

Each pattern-matching clause in case can end with several characters that determine the behaviour of case from that point on. Here are the options to choose from and what they do:

  • ;;: after this pattern matched and commands were executed, exit the case statement here (go to the end at esac)
  • ;&: after this pattern matched and commands were executed, continue with running the commands in the next clause, without pattern matching
  • ;;&: after the pattern matched and commands were executed, try to match the next patterns from here on

For more comprehensive information, refer to GNU Bash Manual: Conditional Constructs

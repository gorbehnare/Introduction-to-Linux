Step 2: Attempting to create the user account and check for success or failure

In order to create a new user from the shell terminal, we use the adduser command. The adduser command is interactive by default (meaning it will ask questions and prompt for answers). Imagine if you are going to be using this script to create 100s of user accounts at a time, reading the user information from a file instead of typing it in the shell. Unfortunately, interactivity is not desired when trying to automate processes. Therefore, refer to the man pages for adduser to find the flags and options needed to use the command properly in one line without the need to interact with the command.

Make sure that the password is disabled upon the creation of the account, and user information (first name, last name and the email address) is also recorded in the user profile for future reference (Hint: see the adduser option called gecos for extra information such as real name and email address).

In order to check to see if the adduser command was successful, we can use the following if statement and take appropriate action in case something goes wrong. Note that the adduser will generate an exit status code of 0 when successful.

# Did adduser work? What to do if there is an error?
# Gives exit status 1 if adduser failed or generated an error
if [ $? -eq 0 ]
   then
      echo "Linux User account for ${FirstName} $LastName was created successfully."
   else
      echo '!!! User account was not added. AddUser Returned exit status '$?' !!!'
      exit 1
  fi

Step 3: Generate a random password

We are going to be generating an RSA SSH key for this user to be able to SSH into our server. However, for better security, we would like to have a passphrase on the RSA key as well.

There is a random number generator device in Linux (/dev/urandom) that we will use for this purpose. We must translate (tr) the output of the random number generators to ASCII letters, numbers, and symbols, then only keep the desired number of them to limit the length. For the sake of time and scope of this lab, we are not going to research this. Below is the command that accomplishes this. Notice the `(back quotes) and ;. You can also try inputting this command into your shell to see what it gives you, and subsequently, modify the command or play with various parameters to learn and explore.

# Generate a strong, 16-character, random password
Password=`< /dev/urandom tr -dc _A-Z-a-z-0-9\%\&\*\$\#\@\! | head -c${1:-16};echo;

Step 5: Generate RSA keys, and check for errors

To give users SSH access to our server, we need to generate an RSA key pair (public and private key). We will need to give the private key, username, and passphrase to the user. The public key will need to be stored in the /home/${username}/.ssh/authorized_keys file.

First, use the man pages for the ssh-keygen command to find out what flags you will need:

  • Specify a location for the files to be stored (we want the key to be stored in ./$username/id_rsa)
  • Specify a passphrase
  • Do not prompt us and do not show output to the console since we do not need to see this

Once you have the command with appropriate flags constructed, you can again have an error handling if statement as follows. This time, we are giving an exit status 2 for our script in case ssh-keygen fails:

# Did ssh-keygen work? If it fails, delete the user account and home directory, and give the exit Status 2
if [ $? -eq 0 ]
   then
      echo "RSA key was generated successfully"
   else
      echo '!!!!!! RSA Key was not created, ssh-keygen exit status '$?' !!!!!!'
      echo "Undoing the changes made to the system so far"
      deluser $username && rm -r /home/$username
      exit 2
fi

Step 6: Move the public key to the appropriate location

We need to use the mv command to move the public key to the user’s .ssh/authorized_keys file, so the user may log in via SSH. You may notice that by default, the ~/.ssh directory may not be created. We do have the choice to create a .ssh directory in /etc/skel when we are configuring our own server, and the adduser will copy that directory structure to every new user’s home directory on that specific machine. However, we may run this script on any system, so we cannot be sure the .ssh directory exists in /etc/skel in every system. Additionally, we can deal with this in our own script without having to depend on the contents of /etc/skel.

Create code that does the following:

  1. Create an IF statement that checks to see if the/home/${username}/.ssh directory doesn’t exist, and create the directory with appropriate permissions

  2. Then move the public key to /home/${username}/.ssh/authorized_keys file

NOTE: Don’t forget to use the chown command to give ownership of these files to the user and their group every time you create a file under the user’s home directory. Otherwise, the files are going to belong to root:root and will prevent the user from accessing or modifying these files.

Testing your script and cleanup

You can simply run your script to see if it works. You can also run individual lines or even multiple lines of commands in your shell to make sure your syntax is correct. However, chances are you must do a lot of trial and error to find all the bugs and to make sure the script is working properly. That may leave your system with a lot of user accounts and a ton of unwanted files under /home.

While testing, if your script does something that is unintended, you may have to undo the changes manually. On the other hand, you can create another simple script, such as the following, to clean up your system and to undo the changes that have been made by your main script while testing:

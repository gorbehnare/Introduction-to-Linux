12 - Linux Software Packaging Formats

Linux is a file-based OS, so one does not need to "install" software as in installation in Microsoft Windows systems. As long as all the files and dependencies are available, an executable can be run from anywhere in the filesystem. The user executing the file must have the "execute" permission for that file. You can obtain the pre-compiled binary, place it anywhere in your filesystem that you have access to and execute.

Running an executable binary is not a viable long-term solution and is certainly not the recommended way to administer a system. How do you know what version of the software is available? How would you keep track of all the files? How would you keep track of all the dependencies required to run this file? How about managing all the manually linked paths, environmental settings, and configuration?

There are many options for obtaining and running software on a system. Here we discuss some of these options.

Compiling from Source

Open-source software is distributed in the form of source code. You can simply obtain the source code along with the appropriate libraries, dependencies, and a compiler application, and compile the source code into an executable binary for your specific system. It is also possible to "cross-compile" source code on one architecture for another architecture. As an example, you can compile code on an AMD64 system for an ARM64 target or any other target where compilers exist. Cross-compiling is often necessary for engineering applications where the target systems do not have the compute capacity to perform code compilation on their own (for example, if you are compiling for a target microcontroller or IoT devices).

However, there are some issues with running pre-compiled and source-compiled applications on a production system such as a server or desktop. The issue remains that there needs to be a way to keep track of all the software versions, where they are located, and manage all of their dependencies and various necessary configuration changes to the system. If there are very few applications, maybe one can resort to pen and paper to maintain documentation for a table of files, versions, and a dependency matrix, but realistically, that is impossible to do in a regular general-purpose system due to the sheer number of files and applications that are needed in a modern system.

Source code is delivered in tarball format. Tarball is an archive file created using the tar utility and then compressed using the gz (Gunzip) utility, with the file extension .tar.gz.

Inside the compressed archive file, developers usually provide specific instructions on how to build the packages on various systems (such as including a readme file). They may also include special scripts (make and make install) to make the process easier for users. Manually building a package may require you to install compiler applications, obtain all the library dependencies, and potentially manually provide the paths and locations for where the files will be placed and where the dependencies are.

Some Linux distros provide source code using their repositories, and a few Linux distros provide additional utilities and tools for users to make obtaining and compiling software easier. As an example, Gentoo Linux provides the Portage package manager, which includes the emerge utility. Using emerge, the user can download the source code from the repository and compile it for their specific system in one step. Additionally, the package manager can keep track of versions that were obtained and compare them to the latest available versions, and allow easy removal of the software if desired, along with many other advantages compared to manually managing software (as explained in the next section).

Building from source code has some advantages:

  • Efficient and small download sizes: source code is mostly plain text, and text compresses well. Depending on the application, other files such as databases and images may be included.
  • Highest efficiency and performance since the application can be specifically compiled for a target system. The compiler can be configured to optimize the binary code for your specific machine with a specific hardware configuration rather than generating generic binaries for any machine.

Package Managers

Package management software and various packaging formats have been created to deal with managing software. Package managers have the ability to:

  • Obtain software from various repositories (sources), such as online repositories, local or network sources, and even from removable media
  • Maintain a database of all available software from all repositories, and what is exactly installed on a system
  • Maintain a database of all files and configurations belonging to a software package
  • Standardize installations, keep the software organized within the filesystem, and apply configurations and security settings consistently
  • Provide tools and facilities to resolve software and dependency conflicts
  • Automate installation, configuration, updates, upgrades, and removal of software packages

Package managers came much earlier than the modern "app stores"; however, in my opinion, most modern "app stores" in proprietary systems such as Microsoft, Google, and Apple are modelled after package managers.

Package Repository

Package managers deliver pre-compiled binary files targeted for your specific system architecture. They obtain packages from a package repository (or repo for short). Different Linux Distros may have various options for package distribution available to them. You have control over what package repositories to use. Some developers submit their packages to the official Linux distros' package repository. Some distributions allow the community to contribute software to a repository they maintain. Some developers or communities maintain and host their own repositories targeting specific distributions.

In short, there are several general categories of software repositories:

  • Official or 1st party repo (most likely the software is tested and thoroughly vetted by the Linux distro)
  • Community repo (more likely that the software is vetted, but the software is maintained by community members, not the entities behind the Linux distro)
  • 3rd party repositories (hosted by 3rd party and do not have any relation to the Linux distro)
  • Self-hosted repositories (you can host your own)

It is important to understand, choose, review, and manage repositories correctly, since the availability of software packages and versions, as well as the speed of updates and patches, depend on the selected package repository and how and by whom it is maintained. Additionally, you must completely trust the proprietors of the package repositories you are using on your systems. Malware can and has been delivered through repositories. This is referred to as a supply-chain attack.

Package Manager and Package Formats

Package managers require software to be packaged in specific formats. Each software package must include metadata such as:

  • a summary, and a description to be presented to the user
  • a list of files contained in the package
  • the version of the software it contains as well as the release number of the package,
  • Author information (when, where and by whom it was built)
  • The architecture has been built for. Common architectures include i386 (old Intel x86), amd65 ( x86_64), armhf(32-bit ARM, such as early Raspberry Pi SBCs), arm64 (64-bit ARM), PA_RISC (the Intel Itanium), mips, etc.
  • checksums of the files contained in the package for integrity checking
  • the license of the software it contains
  • dependencies (which other packages it requires to work properly)

Most Linux distros will include a default package management software. Here are examples of what is included with some popular distros:

  • Debian and Ubuntu:
    • dpkg (Debian package manager), and apt (Advanced Package Tool)
      • apt is a front-end for dpkg (uses dpkg on the back-end to perform operations)
    • use .deb format packages
  • Fedora, Red Hat and Amazon Linux:
    • yum (Yellowdog Updater Modified), and dnf (Dandified YUM),
      • Both yum and dnf use the rpm (Formerly RedHat Package Manager, now RPM Package Manager) on the back-end
      • rpm lacks many features, such as dependency handling
      • dnf is a newer rewrite of the application and is replacing yum
    • Use .rpm format packages.
  • SUSE Enterprise and OpenSUSE:
    • zypper (Zen / YaST Packages Patches Patterns Products)
      • YaST is a GUI and TUI (Text User Interface for terminals) control panel application that, among many other features, uses zypper in the back-end for handling software management
    • uses .rpm format
  • Arch Linux, Manjaro, and other Arch-based distros:
    • Note that Arch is not usually used by enterprises and businesses due to potential licensing complications (may include non-free and proprietary code and binary blobs); however, there is a highly vocal and enthusiastic community around these distros.
    • pacman package manager, uses .pkg.tar.zst package format
  • Gentoo Linux, and BSD (Unix), (historically was on macOS, but is no longer active since macOS X):
    • Portage: an application written in Python, intending to be portable and agnostic, to be able to run on any system
    • Packages contain the source code. Gentoo Linux exclusively deals with source code by default
      • Installing binaries is supported, but not enabled by default
    • emerge is the user-facing CLI utility for obtaining, compiling, installing, and later managing (updating and removing) applications.
    • On the back-end, it uses ebuild, which compiles and builds source code into binary
    • uses .tbz2 format (tar with bzip2 compression). Pre-compiled binary requires additional metadata to be included.

Other Software Package Formats

As you might have noticed, you can use a package manager to install an application binary natively on a Linux-based OS. However, these binaries must be formatted for the specific package manager. Different systems have different package managers, so a developer must package their software for many different distros in different formats. Therefore, there was a need for software formats that can run universally on any Linux system.

The "Running on any Linux system" requirement has many complications. Firstly, it means that the package must account for any of its dependencies in a portable way. As an example, different applications may have a dependency on different versions of the same library, while two versions of that library cannot be installed at the same time, and the user needs both applications to be installed in the system. Secondly, just downloading applications and installing them on your system and replacing files may cause changes to your system that may cause damage. Security is another issue, because a natively installed application can potentially access things it may not be required to access.

Many solutions have been proposed to address any of these problems over the years, and currently, there are a few popular software packaging formats that have come to exist. Namely, the snap, flatpak, and appimage formats for software distribution.

Appimage

The appimage format is a binary file that includes the entire package in one file. It includes the application binary, all the required libraries, and any other runtime dependencies packaged in one file. This makes appimage files portable. You can simply place this single file anywhere and execute it. Since the file is self-contained, it can potentially save user configuration and other information inside the appimage file itself (this depends on the application and its design), so you can simply move your application from system to system, and all your custom configuration will follow.

  • There are no repositories for appimages; you simply download the file from the internet. There are internet sites that catalogue and host these files, such as AppImageHub.com, but there is no official or authoritative location for all appimages.
  • Applications running from an appimage cannot be updated. You must manually go to the source and check for new versions. Additionally, you cannot do partial updates, so you must download an entire new .appimage file.
    • There are 3rd party applications (not discussed here) that attempt to help you manage your appimage files, and some developers attempt to make the appimage application updateable, but these are all optional
  • The appimage files can get very large, because they are self-contained
  • Applications running from an appimage are not aware of the host system configuration and settings, do not follow the system theme and colours
  • Mostly, GUI applications are shipped in this format.

Snap

The snap packages are created and maintained by Canonical (the parent company of Ubuntu). The .snap files are self-contained binaries that execute in a sandbox. Running snap requires the installation of the snapd service on your system. Currently, snap ships with Ubuntu. Although it can be installed in a few other distros, it is not widely used and does not come installed by default. CLI and server applications can be shipped in snap format as well.

Some advantages make snaps suitable for server and enterprise applications:

  • Snaps are sandboxed and run from a virtual volume, and usually they will not get access to the rest of the system (good for security controls).
  • Updates can be delivered as "delta updates" (only binary differences are downloaded), which reduces network bandwidth requirements, which is important to enterprises, having implications of tens of 1000's of servers in a datacenter attempting to download updates.
  • Snaps can be updated live, without stopping the application or service (important for availability). After the update is delivered, the snap can be restarted, and will switch to running the latest version

There are several issues and cons with the snap format that make it unsuitable for all situations:

  • Currently, snap is controlled by Canonical, and cannot make use of 3rd party repositories. The SnapStore and Snapcraft repositories are controlled by Canonical
  • Mainly available on Ubuntu by default, can be installed on other distros, but requires additional services to be installed
  • snap files can be larger in size, because they contain the application dependencies
  • Since snaps are sandboxed, they are not able to access system configuration and resources. They have an administrative overhead of managing permissions to allow a snap access to certain directories or files, or have to deal with complicated paths inside virtual volumes for storage
    • This also causes snap applications to be slower to start
    • In GUI, in addition to being slower to start (at least for the first time after boot), they are unaware of system themes and configuration, and users may find accessing their files and data through the sandbox challenging

Flatpak

Flatpak is a package management utility for Linux, which manages packages in the .flatpak binary format. Flatpak applications are sandboxed in partial isolation from the rest of the system, and are suitable for desktop and GUI applications.

  • flatpak applications are fast, run almost as fast as native binaries for most applications
  • dependencies are handled by the flatpak utility, and multiple versions of a dependency can co-exist as needed
  • The flatpak utility can obtain software from any repositories called "remotes".
    • Currently, flathub.org is the most popular "remote"; however, anyone can host a "remote," and you may add other software sources to your system.
  • Partial downloads for updates are supported.
  • The flatpak utility can potentially be usable on other Linux-kernel-based systems, such as ChromeOS and Windows, using Windows Subsystem for Linux
  • The flatpak utility provides a permission system, where application access to system resources can be managed per application, or as a bulk.
    • These permissions can be managed from CLI; however, there is a GUI utility available called flatseal that provides a GUI for permissions management if desired.

There are a few potential disadvantages and issues with using flatpak:

  • developers specify dependencies with their packages. It is up to the developers to update their application and their dependency requirements. If they neglect to do so, the system may have to hold on to outdated, unmaintained, or vulnerable versions of the dependencies
  • flatpak applications can potentially be used for storage compared to natively installed counterparts, due to applications bringing their own dependency and libraries that can be duplicates

Immutable Systems

There are immutable systems, where the OS and software are designed to be unmodifiable (read-only). The argument in favour of immutability is that malware cannot be installed, and the system cannot break since no fundamental changes are allowed to be made to the system. This can potentially improve reliability and reproducibility, especially useful in very large environments (such as cloud deployments), mission-critical systems, and high-security environments, or in systems that are expected to function for a very long time without changes or deviation.

There are various methods of implementation and achieving immutability, from using fixed images burned into a ROM or Flash storage and the use of special-purpose hardware, to implementing immutability in the software on general-purpose hardware. User configuration and customization, if required, are stored separately or require other advanced architectures and technologies such as sandboxing, virtualization, or containerization.

Examples of immutable systems based on the Linux kernel are ChromeOS and Android. In datacenters, you may find embedded systems such as the Dell iDRAC (integrated management computer in Dell server) and HPE iLO and Onboard Administrator (Integrated management computer for HPE servers and for server blade enclosures, respectively). Other examples include systems delivered as network appliances (routers, switches, firewalls and security appliances), and commercial proprietary systems such as SAN and NAS systems (for example, HPE 3PAR and Hitachi SAN Storage systems). IoT and embedded devices are also immutable. Systems used in automotive systems, such as control computers, infotainment systems, and gauge clusters, are immutable by design. These systems do not offer any type of package management at all. They come with all the software they need compiled and built into the system.

In addition to special-purpose hardware and appliances, immutable Linux Desktop OSes exist that can be installed on your general-purpose PC. There are various use cases and target audiences for these. Some examples include:

  • Fedora Silverblue and OpenSuse MicroOS generally make the root partitions read-only, and rely on Flatpak, Appimage, and Snaps to provide required software and functionality to the user.
  • In NixOS, a configuration file is used to create the entire operating system, including all the services, configuration and software. The user maintains the configuration file and can replicate the entire system in a "reproducible" manner (the same configuration results in the exact same system).
  • Vanilla OS offers a baseline read-only operating system, where containers are used to run any other Linux distro (such as Arch, Fedora, OpenSuse, etc.) and run the desired applications from within the container. This has the advantage of potentially being able to access any software on any Linux distro.
  • SteamOS on the SteamDeck is another example of a loosely immutable system, where, despite SteamOS being based on Arch Linux, a general-purpose OS, and the hardware being just a general-purpose AMD64, access to the standard package manager is restricted within the software. Any changes made by enabling a package manager are reverted during the next SteamOS update.

In the datacenter and cloud environments, many products perform "atomic" operations when launching or updating services. Examples of two products are Alibaba Cloud's ContainerOS, and Red Hat Atomic Host. Both of these use containers to launch services as immutable software. When software is updated, the entire container is replaced with the new version in an all-or-nothing fashion. There is no patching or updating individual application files inside the containers. This provides immediate rollbacks in case of failure without any service disruption, and also allows for testing and validating complete images without having to micro-manage individual configurations and files for each instance, when deploying thousands or hundreds of thousands of instances.

10 - System Admin Utilities

This section provides useful utilities and background information for system administrators.

Time and Date

System Local Time

Most operating systems (except Windows) expect your computer's real-time clock (RTC) to be set to the UTC timezone. Linux systems then apply an offset to the UTC to display the local time zone to the user. This system has the advantage that each user gets to see system time in their own timezone, regardless of where the server is located.
Noteworthy system configuration files:

  • /etc/timezone: contains the system default timezone variable
  • /etc/localtime: From man 5 localtime:configures the system-wide timezone of the local system that is used by applications for presentation to the user.
    • Links to /usr/share/zoneinfo
    • If this file is missing, the UTC timezone is used
      To manage your timezone or reconfigure the date and time, you can use the following methods and utilities:
  • Using dpkg-reconfigure tzdata is the preferred method in Ubuntu and Debian to configure the default system timezone
  • Use timedatectl set-timezone <timezone> if you are using other Linux distros. You are required to know valid and correct values for the <timezone>, which are called the "proleptic TZ string" (see man 5 tzfile for more information)
  • The tzselect utility to view and select a timezone for your shell. It has a menu-driven interactive mode where you get to select your timezone based on your location, without knowing the "proleptic TZ string" format
  • Use date to display the current date and time in various formats.
  • The hwclock is used to administer your hardware clock time. In addition to setting the hardware clock and viewing the current time and date, you can perform operations such as setting the system clock from the hardware clock or vice versa.

Network Time Protocol (NTP)

If you are maintaining a network of systems and IoT devices, it is crucial to ensure that all systems have the correct and accurate date and time at all times. The accuracy of system clocks can affect anything from logging and monitoring to file backups and data integrity.
Most servers and IoT devices can rely on the NTP to synchronize the system time and date with one or more NTP servers. Public NTP servers are run and maintained by various parties over the internet. Some NTP servers are maintained by governmental organizations, while others may be operated by standard bodies or private corporations and individuals.
If you are in charge of creating or maintaining a network, there are many reasons you should configure an NTP server for your local networks. This is especially useful when your network architecture uses segregated networks where systems (for example, IoT devices) are not allowed to access the internet.

  • For more information about the NTP protocol and the foundation in charge of maintaining NTP, refer to ntp.org.
  • The ntpdate utility (see man 8 ntpdate) is used to set the local date and time from NTP servers
    • /etc/default/ntpdate is the default configuration file when this utility is used for time synchronization
    • /etc/ntp.conf is the daemon configuration if the ntpdate utility is going to be running in the background to perform synchronization
  • The chrony NTP client/server utility is another alternative (see Ubuntu Server Documentation: Synchronize time using Chrony for more information).
    • chrony is the default utility in Red Hat and SUSE Enterprise distributions
    • This utility is stated to become the default NTP client in Ubuntu in future releases (as of version 25.10 and beyond), but has been added to the repository for some time (you can install this now)
    • The biggest advantage is support for the NTS protocol (Network Time Security) to prevent spoofing attacks
  • The timedatectl utility is used to view and control the system time and date.

Host Identification

The hostname command can display your current system hostname.
In Ubuntu/Debian-based distributions, hostnamectl can be used to display a more comprehensive system information. The hostnamectl is part of the larger and comprehensive systemd. It can distinguish 3 different hostname types for a machine:

  • Pretty name: user-defined and can contain special characters.
  • Static name: user-defined, and is what the back-end systems use to identify this machine. This name has limitations (single word with no spaces, small letters, no special characters, etc.).
  • Transient name: If a static hostname is not defined, a fallback value is generated. This is often used for large virtual and cloud deployments.
    For more information, please refer to man hostnamectl.

Your system's static hostname is defined and stored in the following locations:

  • /etc/hostname
  • /etc/hosts

In Ubuntu Server, the transient name is managed and configured via the cloud-init set of tools. The configuration file is located at /etc/cloud/cloud.cfg. As an example use case, if you are creating a master template image for virtual machine or cloud deployments, you can configure this file to specify if you would like to preserve the hostname or not via the preserve_hostname: true/false line.
For more information, please refer to man cloud-init or the Cloud-init documentation.

Memory Management

You can view the current memory usage and status of your system using the free utility. Here is a sample output:

user@system:~$ free -h
               total        used        free      shared  buff/cache   available
Mem:            29Gi        11Gi        12Gi       2.2Gi       8.9Gi        18Gi
Swap:          8.0Gi          0B       8.0Gi
user@system:~$ free -h
               total        used        free      shared  buff/cache   available
Mem:            29Gi        11Gi        12Gi       2.2Gi       8.9Gi        18Gi
Swap:          8.0Gi          0B       8.0Gi

Your system has two general types of memory: mem and Swap

  • The mem from the example above refers to the RAM (Random Access memory, which is also referred to as the physical memory). This corresponds to the actual available random access memory hardware that your system has access to.
    • The free amount refers to available unused RAM.
    • buffered/cached memory is in RAM and refers to data that is loaded to improve performance and system responsiveness. This data is usually a duplicate copy of what exists on disk, so it can be freed if needed.
    • The available memory is the total RAM that can be used by your system right now, including the shared cache buffers and caches that can be potentially freed if needed.
  • The swap memory, which is also sometimes referred to as virtual memory.
    • Swap memory is stored outside of the physical memory to free up the RAM, so that your system can deal with physical memory limitations. Storage location for swap can be:
      • swap file in your filesystem
      • swap partition, which is a dedicated storage disk partition created for this purpose
    • swap memory is used when:
      • Your system is loading large amounts of data that do not fit into RAM
      • Your system wants to preserve a cached application data, but wants to free the RAM for other applications to use when that application is "sleeping" (hence the term "swapping" it out of RAM)

You can configure and control your swap memory as desired or needed by your specific system, by allocating swap partitions on a destination storage media (such as HDDs, SSDs, and dedicated SSD hardware used as cache or buffer in some systems).

  • Use lsblk and fdisk utilities to view and manage any swap partitions
  • Use the fallocate utility to allocate space in a filesystem for swap, and then use the mkswap command to create a swap file
  • Use swapon and swapoff commands. The swapon command also displays your current swap information. Below is the output of the swapon command, showing the location and size of the swap file:
user@hostname:~$ swapon
NAME      TYPE SIZE USED PRIO
/swap.img file   8G   0B   -2
user@hostname:~$ swapon
NAME      TYPE SIZE USED PRIO
/swap.img file   8G   0B   -2

Comparing Files

  • The cmp utility can be used to compare files using a byte-by-byte binary compare. This is useful for verifying if two files are exactly the same or not (for example, if you want to know if your backup is exactly the same as your source, or if the file has been modified).
  • The diff utility is used to display the difference between two text files. The diff <file1> <file2> will output the line numbers and columns and display the line in the file on the left (using <), and the line in the file on the right (using >). You may often run into the diff utility during system upgrades when the installer application will prompt you if you would like to change a configuration file in your system to a new version. The installer will often give you the option of viewing the diff so you can identify what the differences between the old and the new content are, and make a decision.

System Processes

A process is the code, data, and control information for a program as it is executed on a system. The control information is stored using a PID (process ID) in the kernel's process table. The process table contains the uid (user ID), the gid (group ID), and the environmental variables such as the path to the executable file, memory locations, and other information.
The uid and gid determine access to resources via the file permissions model. As a very simplified example, if a user is a member of an lp group, they may be able to write to a printer device file via the group's permissions on that device file, effectively controlling whether they can print or not using the gid.
Simple utilities and commands related to processes:

  • ps: report a snapshot of the current processes.
  • pstree: displays the hierarchy of the processes (parents and children) in tree format
  • kill: (see man kill) send a signal to a process. The default signal is TERM (terminate), but you can send other signals such as STOP, CONT (continue if stopped), CORE (terminate and dump core for troubleshooting), etc. See man 7 signal for an overview.
  • top, and variations and many alternatives, such as htop, slabtop, atop, iotop, iftop, and gtop, and many others, for viewing and managing all system processes, and other performance metrics.

User Management

  • The /etc/skel: When installing a system that will be used by many users, you can control the contents of the home directory for all users by modifying the files under /etc/skel.
    • The adduser utility will simply copy everything from /etc/skel into the new users' home directories and change the owners of the file to the new user.
    • You can control many things, from what default files and directories are created (for example, Documents, Downloads, Pictures) to their application configuration (for example, add certain bookmarks and configurations to their browser) and their bash environmental variables.
      • include custom .bashrc, .ssh, .config, etc.

System User and System Groups

In Linux, all user accounts are the same. There is no difference between a "system" and a "human" user account. However, each distribution has certain conventions for differentiating between system and regular users and groups.

A simple example use case for differentiating system and regular users is determining if the user is going to be allowed to authenticate into a shell. You would want your regular users to have a default shell (for example, bash or a GUI shell), but your lp user (user account or group reserved for the printer device or service) does not need and should not have access to authenticate via a shell. This will have many security complications. Another example of the need for differentiation between system and regular users is the fact that your system uses the same mechanism to know which users to display as selectable on a GUI login screen and which ones to hide.

When you issue the adduser command, you can specify (via flags and options) if you are going to create a regular user (default), or a special system user. In Debian and Ubuntu-based systems, by convention, UIDs and GIDs below 1000 are reserved for system users and groups. You may have noticed in the previous section that the first user that is created when you installed Ubuntu Server is usually uid=1000, gid=1000. Your root account is uid=0. Additionally, there are many Linux user accounts and groups that come pre-configured on your Linux server.

Debian documentation has a set of default SystemGroups that you may find on the Debian Wiki, SystemGroups. Some of these groups have associated users, and some do not have any associated users by default. These groups are useful for bulk managing user permissions for various services and facilities without having to dig into your filesystem and manually changing owners and permissions for devices, applications, and services.

Some examples from the Debian Wiki, SystemGroups documentation include:

  • lp (LP): Members of this group can enable and use printers. (The user lp is not used anymore.)
  • lpadmin (LPADMIN): Allows members to manage printers and pending jobs sent by other users.
  • scanner: Members of this group can enable and use scanners.
  • tty: TTY devices are owned by this group. This is used by write and wall to enable them to write to other people's TTYs, but it is not intended to be used directly.
  • staff: Allows users to add local modifications to the system (/usr/local) without needing root privileges (note that executables in /usr/local/bin are in the PATH variable of any user, and they may "override" the executables in /bin and /usr/bin with the same name). Compare with group adm, which is more related to monitoring/security.
  • shadow: /etc/shadow is readable by this group. Some programs that need to be able to access the file are SETGID shadow.
  • utmp: This group can write to /var/run/utmp and similar files. Programs that need to be able to write to it are SETGID utmp.
  • disk: Raw access to disks. Mostly equivalent to root access.
  • wheel: (Useful for Unix compatibility) This group can be created to alter the functionality of the module pam_wheel in /etc/pam.d/su to restrict the execution of su. If this group does not exist, only members of the root (usually the root itself) are affected. By default, this group does not exist, and all the configurations with pam_wheel are commented in /etc/pam.d/su. See the page about the wheel_pam module for more information.
  • libvirt: Users in this group can talk with the libvirt service via dbus, as defined in /etc/libvirt/libvirtd.conf
  • kvm: Users in this group can use the KVM acceleration of virtual machines.

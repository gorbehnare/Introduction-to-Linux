07 - Software Management

Running Any Software

In Linux/Unix, there is no “Windows Registry” or “add/remove programs” to keep track of where the programs are located, where to put shortcuts, or what files, libraries, or other dependencies are required.

To run any software, you can obtain the software source code and compile it on your system to generate the executable binary, or you can obtain the pre-compiled binary for your system architecture. Then you can run the executable directly on your system from anywhere
that the user has execute permission. You can chmod and add execute (+x) permission on any file that you have write access to.

On the other hand, you are left to manually check to ensure that all dependencies for the program are met, including all necessary libraries, other programs, and utilities that the software requires, are present in your system. Additionally, you are left to manually manage and maintain files for your applications. For example, how do you go about figuring out which version of these files you have installed, and how do you update them?

Package Managers

Package managers automate managing your application. Package managers can download the software files, place them where they are designed to go in your system, keep track of and update them, and cleanly remove them when you no longer need them. Package managers maintain a local database of all managed programs and applications in order to keep track of all files and their versions.

Package managers require the software and its components and files to be packaged in a specific format. These specifically formatted files are called packages. Packages can contain all files of specific software files such as source code or pre-compiled binaries, for specific systems and architectures. Generally, the packages may include a list of recommended and required dependencies, as well as installation scripts for specific operating systems.

Some examples of software package formats are .deb, .yum, .rpm, and .snap. These package formats are specifically designed to be used with their corresponding package management software.

Software Repo (Repository)

The easiest metaphor that can be used these days is to compare software repos to modern app stores. Software repos have been around for much longer than the proprietary app stores from Microsoft, Apple and Google, but they serve a similar function. They are used to publish, store, and distribute software packages.

Repositories can be stored in various locations, from local storage media such as CDs or DVDs to network shares to the internet. Various entities, from individuals to larger organizations, are free to create and maintain their own repositories. You can add additional software repositories to your supported Linux system in order to access additional software that may not have been included with your Linux Distro, for example, to access a developer-maintained repo to receive the latest or development version of a software, you may add that developer's repository to your package manager.

Some package repositories also offer their own package management system for developers and end users. Developers will have the required tools to publish and maintain their own packages in the repository, and package managers from users' systems can obtain the metadata from the repo to keep their local databases up-to-date on what software and what versions of software are available.

Here are some examples of software repositories for an Ubuntu system. Canonical (Ubuntu's parent company) maintains and supports the following repositories for Ubuntu:

  • Universe: Community-driven, Open-source packages, included with Ubuntu
  • Multiverse: Community-driven, mostly open-source but with restrictive licenses preventing them from being included in a free-libre OS
  • PPA: "Personal Package Archive" provide infrastructure for 3rd party developers and contributors to distribute their own packages

Additionally, some developers and organizations maintain their own repositories instead of using Canonical's. Some examples:

  • Brave-Browser (see https://brave.com/linux/)
  • Firefox (see Install Firefox .deb package for Debian-based distributions)
    There are many more examples, and some repositories can be free, while others can be exclusive or pay-to-access.

Package Managers in Ubuntu

For native software installation:

  • APT (Advanced Packaging Tool)
    • Interactive, easy to use, and your default go-to!
  • dpkg (Debian Package Manager)
    • Backend system used by APT, useful for specific advanced tasks and scripting
  • See: https://documentation.ubuntu.com/server/how-to/software/package-management/index.html

    • Requires a service called snapd (snap daemon) to manage and control snaps

Using APT

All APT configurations are located under /etc/apt. The files containing the definition of package repositories are located under the /etc/apt/sources.list.d directory. You can add or remove repositories by adding and modifying .list and .sources files in the directory.
As of APT version 3.0, which is included with Ubuntu 24.04 and newer, .sources files are being used, and the use of .list files is being deprecated.
Here is an example of a Microsoft VS Code repository for Linux in both formats. Please note that VS Code is a GUI tool and is not intended to be installed on a server:

# Microsoft VS Code (the old "vscode.list" file)
deb [arch=amd64 signed-by=/usr/share/keyrings/vscode.gpg] https://packages.microsoft.com/repos/code stable main
# Microsoft VS Code (the old "vscode.list" file)
deb [arch=amd64 signed-by=/usr/share/keyrings/vscode.gpg] https://packages.microsoft.com/repos/code stable main

Same repository definition converted to the new .sources format:

### THIS FILE IS AUTOMATICALLY CONFIGURED ###
# The new "vscode.sources" file
Types: deb
URIs: https://packages.microsoft.com/repos/code
Suites: stable
Components: main
Architectures: amd64,arm64,armhf
Signed-By: /usr/share/keyrings/vscode.gpg
### THIS FILE IS AUTOMATICALLY CONFIGURED ###
# The new "vscode.sources" file
Types: deb
URIs: https://packages.microsoft.com/repos/code
Suites: stable
Components: main
Architectures: amd64,arm64,armhf
Signed-By: /usr/share/keyrings/vscode.gpg

Both files contain the same information; however, they are formatted slightly differently. There is an apt modernize-sources function that can convert existing old format to the new format on systems that are upgraded and still contain old formatted .list files.

APT Common uses

  • Update the local database with the latest meta-data from repos:
    • sudo apt update
  • Apply the upgrades to software:
    • sudo apt upgrade
  • Install software:
    • sudo apt install <package>
  • You can also:
    • sudo apt search <package>
    • sudo apt list <package>
    • sudo apt remove <package> or sudo apt remove --purge <package> to remove all residual configuration files as well

Using dpkg

The dpkg utility is the back-end package manager in Debian-based systems, including Ubuntu. APT and other front-facing utilities are easier-to-use utilities that call dpkg on the backend to accomplish their tasks. However, dpkg can provide advanced functionality when administering or troubleshooting a system, or simply trying to create scripts that may run in multiple distros. Here are some examples of dpkg uses:

  • List all packages in the local database, installed or not:
    • dpkg –l or use dpkg –l | grep <keyword> to filter for specific keywords
  • List all the files a package has in a system:
    • dpkg –L <package>
  • What package uses this file or put this file here:
    • dpkg –S <file>
  • Install a manually downloaded .deb package file:
    • dpkg –i <file.deb>
  • Re-run the installation script or installation wizard if interactive:
    • dpkg --reconfigure <package>
